How to make a website from scratch and still survive (somehow)

So this website is up finally, after long tribulations. I thought about making a celebratory post to talk a little about what happened in the making of this thing, since I think it will be useful to someone, out there.

"Programming" a website

Okay, so I wanted to write my own static site generator, for a handful of reasons.

The biggest reason is probably because programming a website sounds simple. It sounds simple, that's important. And since I've done "real" programming in "real" languages, that sounded like a fun little project to tackle.

Boy, was I underestimating the task.

I could have used software like Jekyll to generate my website, but I found that it was quite hard actually to make it do what I wanted to do. I would have probably spent as much time making my website in jekyll than making it from scratch, and I wouldn't have been really happy with the result anyway.

So instead I decided to use the hackiest of languages: bash. That's mostly because I underestimated how hard it would actually be to generate a whole website like this. In the end, it wasn't super bad, but it was difficult enough that it made me question my choice a couple of times along the way. I'd give it two thumbs up and one down in terms of difficulty.

What I like about bash is that it is the same environement as the terminal, which means that translating stuff from one script to another would be as simple as exporting variable and stuff. And having everything in an omogeneous mass of the same language is for sure something that makes it easier to work with. The only drawback that I had is that the syntax can become quite horrendous, especially when using arrays, but that doesn't influence the final product so I think it's more of a pet peeve of mine more than an actual problem.

The way it works is: a script puts all the basic html for a page together (navigation, structure, etc). Then there are many smaller script specific to each page that generate the data for each page and implement it back in the base structure. It's all automatic, and it allows me to regenerate everything from the bottom up in a fraction of a second.

Yep, that means that even this article is actually a snippet of HTML code inside of a bash script. Not super elegant, but it works. And writing HTML code by hand isn't super time consuming, especially by using some automation tools.

The only problem is that the code is quite hard to debug. If some professional web developer saw my code he would probably get a stroke, but since this isn't intended to be a huge site it works quite well, actually. And since it's mostly variable substitution and string manipulation it's never too hard to pinpoint where a problem arised. I can add more stuff quickly now and if I need to change something or add a new page I can just edit some of the scripts I edited.

Do I reccomend this path, though? Not really, no. I did this mostly because I'm interested in data generation and I thought it would be a fun experiment, so for me it would have been a win even if I didn't get anything out of it. But if you want to do a personal blog I highly reccomend using already existing and estabilished static site generators. That can save you a headache and a half.

Talking the language of Google

Search Engine Optmization (SEO for short) was one of my priorities. I realised that most art websites are quite bad at this kind of stuff, and I wanted my art and games to be properly recognized by Google and correctly indexed.

The biggest problem with making this website has probably been trying to make the search engines understand what is actually going on on the page. Good old search engine optimization, as they. It looks like that Google crawler is smart, but it is no genius, and it needs a lot of stuff to be pointed out for him in order to understand what the hell it is going on on the page.

The answer to this problem is (mostly) metadata. Google apparently accepts metadata called LD-JSON, which is structured data that you put in your header to make the search engine understand what the page contains, what is relevant and what should be ignored. It's based on JSON, so it's pretty easy to read and edit. The basic schema for this is found at the appropriately named schema.org. Basically you're just trying to tell Google that that thing that looks like an article is in fact an article, and so it can treat it as that. If that metadata isn't present, it has to infer what it is from the context, and even if it's probably smart enough to understand what is going on, I have some doubts that it is giving precedence to stuff that it knows what it is because it has been told him over other stuff that he has to understand by himself.

It may seem like little stuff, but it can actually get quite deep. You can have metadata about the caption of an image that you've used in a news article. You can have the number of reviews for a videogame that you've made. How important this data is, I don't really know. Google however lets us use a tool where you can check if the pages are working fine and if some important data is missing.

I ended up losing a lot of time to catalog all of my art and the games I added to the website according to this data. Again, I don't know if this was just a waste of time or not; but I think it makes things a little easier in the long run. Moreover, once you do this once, you can reuse the same format next time you need it.

A note on responsive design

Of all the things that I thought would take me the most time and instead happened to be one of the easiest things I did, was probably developing the website with a responsive design. If you are out of the loop, that just means programming the page in a way that shows up just fine both on desktop and mobile, instead of needing to have two pages for different devices.

I've worked a little in CSS in the past, but I've never been super serious about it. And, I have to say, it's been a breeze to learn. Sure, it has its own quirks (for example, how it handles vertical alignment), but it's very easy, lightweight and pleasant to use, especially if you keep the structure of the website itself simple enough.

The main problem under this aspect has been programming the navigation menu. It needed to work both on mobile and with a mouse, which meant it needed to open drop down menus in different ways. I ended up programming most of it in javascript, mostly because it's much, much easier to handle this way. The code is extremely basic, mostly just programming the menus to close and open on clicks. This was rather straightforward, especially since I'm used to programming in Javascript from all the ludum dare stuff I've done in the years

(By the way, why not check out Fallen Girl? It's a simple platformer I released back in Christmas. I think you'll like it).

Hosting and Domain

I ended up using Github Pages as my main host. Reason is: it's free, and it's very easy to use. I was never a big fan of FTP, and Git allows to make edits very quickly and easily. Plus, free SSL encryption for the https is always a plus.

I bought the domain at NameCheap, mostly because I used them before without problems, and because it includes free whois protection. It basically prevents bad people from the internet to find out where you live by looking at the data the registrar stores with your domain. I don't think people care about my stuff enough to actually seek out my residence, but I've been on the internet long enough to realize that this kind of stuff can bite you in the ass when you least expect it, so it's better to be safe than sorry.

Final Thoughs

As an internet citizen for more than 20 years now, actually making my own website from scratch like this was an illuminating experience. I gained a lot of respect for all the work that goes even in the more modest sites, and peeking behind the curtains of what happens in making one really fulfilled my nerdy side.
